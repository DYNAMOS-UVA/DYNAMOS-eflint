# eFLINT Policy Layers: Design & Implementation

This document combines the design specification of the eFLINT reasoner interface with its implementation as the eFLINT specification.

---

## Design: eFLINT Reasoner Interface

The contract between the Policy Enforcer and the eFLINT reasoner determines what a policy author can express without code changes (F6). It is structured as **three concentric layers**, modelled on the BRANE-chk pattern:

| Layer | Name | Description |
|-------|------|-------------|
| 1 | Interface Policy | Small, stable spec shipped with the enforcer. Declares base fact types and five query facts (no derivation rules). |
| 2 | Agreement Policy | Runtime-mutable normative spec by a policy expert. Supplies `Holds when` derivation rules for Layer-1 query facts. |
| 3 | Request Facts | Ephemeral facts generated per evaluation by the Request Context Builder. |

### Layer 1: Interface Policy

A small, stable eFLINT specification shipped with the enforcer. It declares the base fact types that mirror the platform's agreement vocabulary (e.g. `requester`, `data-steward`, `archetype`) and **five query facts** that the runtime invokes:

1. `permitted-request` — overall request admissibility (at least one valid steward)
2. `permitted-at-steward` — per-steward admissibility (requester has standing)
3. `valid-archetype` — generative query enumerating admissible archetypes per steward
4. `valid-compute-provider` — generative query enumerating admissible compute providers
5. `permitted-continuation` — ongoing authorisation for in-flight sessions

These query facts have **no** derivation rules in Layer 1; their `Holds when` clauses are supplied by Layer 2 via eFLINT's `Extend Fact` mechanism.

The interface policy also defines the request lifecycle through normative concepts — an act (`submit-data-request`) and a duty (`obligated-log`) — providing a formal basis for auditability (F7).

### Layer 2: Agreement Policy

The runtime-mutable normative specification authored by a policy expert. It must supply derivation rules for each Layer-1 query fact. Because the interface policy fixes the query schema, the agreement policy can be rewritten without touching enforcer code (F2).

In practice, Layer 2 decomposes into:
- **Shared agreement rules** — intermediate facts and `Extend Fact` clauses shared across a consortium
- **Per-steward agreements** — concrete fact instances for each data steward

### Layer 3: Request Facts

Ephemeral facts generated by the Request Context Builder for a single evaluation. The decision and its derivation tree are persisted by the Audit Logger.

### Design Rationale

This three-layer separation embodies the principle: *"By separating the rules formalising the norms from the implementation of a system, the transparency, adaptability and maintainability of the system are increased."*

A **runtime policy update is a replacement of Layer 2 only**, and is admissible only if the new agreement policy still derives all five Layer-1 query facts. The Policy Repository validates this property by submitting a synthetic probe scenario to the reasoner before activating a new agreement policy.

## Implementation: eFLINT Specification

The eFLINT specification realises the three-layer separation. Layer 1 fixes the vocabulary and query facts; Layer 2 supplies the derivation rules that make those query facts hold; Layer 3 is generated per call by the Request Context Builder.

### Layer 1: Interface Policy (`01_interface_policy.eflint`)

The interface policy declares the base fact types mirroring the domain vocabulary already present in the DYNAMOS agreement structure: `requester`, `data-steward`, `dataset`, `request-type`, `archetype`, and `compute-provider`.

It also declares a helper fact `requested-steward` that marks which data stewards are addressed by the current request.

**Query-fact declarations (no `Holds when` clauses):**

```eflint
// Layer 1 — query facts (no derivation rules)
Fact permitted-request
  Identified by requester

Fact permitted-at-steward
  Identified by requester * data-steward

Fact valid-archetype
  Identified by requester * data-steward * archetype

Fact valid-compute-provider
  Identified by requester * data-steward * compute-provider

Fact permitted-continuation
  Identified by requester * data-steward * archetype
```

These are declared as empty shells whose `Holds when` clauses are supplied by the agreement rules via `Extend Fact`.

Additionally, Layer 1 defines the request lifecycle: the act `submit-data-request` and the duty `obligated-log`, embedding the Hohfeldian power-duty relation.

### Layer 2: Agreement Rules (`02_agreement_rules.eflint`)

The agreement rules translate the procedural logic of the mock policy enforcer into declarative normative derivations. Each decision diamond in the flow becomes a `Conditioned by` or `Holds when` clause in eFLINT.

#### Agreement Existence

The first check: does an agreement exist for the requested data steward?

```eflint
Fact agreement Identified by data-steward
```

At runtime, the Request Context Builder populates this fact for each steward whose agreement is found in etcd. If no agreement exists, the fact is absent and all downstream derivations for that steward fail.

#### Steward Capabilities

When an agreement exists, the enforcer reads the supported archetypes and compute providers:

```eflint
Fact steward-supports-archetype Identified by data-steward * archetype
  Conditioned by agreement(data-steward)

Fact steward-supports-compute-provider Identified by data-steward * compute-provider
  Conditioned by agreement(data-steward)
```

The `Conditioned by` clause is the normative equivalent of the procedural guard: any attempt to assert `steward-supports-archetype(VU, computeToData)` without a prior `agreement(VU)` in the knowledge base is rejected by the reasoner.

#### User Relation Check

Is the requesting user present in the steward's relations?

```eflint
Fact has-relation Identified by requester * data-steward
  Conditioned by agreement(data-steward)
```

An instance `has-relation(Alice, VU)` holds when the requester Alice is listed in VU's agreement relations.

#### Relation-Level Permissions

Fine-grained allowances per relation: `requestTypes`, `dataSets`, `allowedArchetypes`, `allowedComputeProviders`:

```eflint
Fact relation-allows-request-type Identified by requester * data-steward * request-type
  Conditioned by has-relation(requester, data-steward)

Fact relation-allows-dataset Identified by requester * data-steward * dataset
  Conditioned by has-relation(requester, data-steward)

Fact relation-allows-archetype Identified by requester * data-steward * archetype
  Conditioned by has-relation(requester, data-steward)

Fact relation-allows-compute-provider Identified by requester * data-steward * compute-provider
  Conditioned by has-relation(requester, data-steward)
```

#### Archetype Intersection

The core logic: only archetypes that appear in **both** the relation's `allowedArchetypes` and the steward's top-level `archetypes` are valid. Expressed as a conjunction:

```eflint
Extend Fact valid-archetype
  Holds when relation-allows-archetype(requester, data-steward, archetype)
    && steward-supports-archetype(data-steward, archetype)
```

This single derivation rule replaces the procedural intersection loop.

#### Compute-Provider Validation

Analogously, valid compute providers are derived by intersecting the relation's allowances with the steward's supported providers:

```eflint
Extend Fact valid-compute-provider
  Holds when relation-allows-compute-provider(requester, data-steward, compute-provider)
    && steward-supports-compute-provider(data-steward, compute-provider)
```

#### Per-Steward Admissibility

Determines whether a steward belongs in `valid_dataproviders` or `invalid_dataproviders`:

```eflint
Extend Fact permitted-at-steward
  Holds when agreement(data-steward)
    && has-relation(requester, data-steward)
```

#### Overall Request Decision

The final decision — `request_approved` — maps to `permitted-request`, which holds when at least one requested steward is permitted:

```eflint
Extend Fact permitted-request
  Holds when requested-steward(data-steward)
    && permitted-at-steward(requester, data-steward)
```

### Backward Compatibility: Legacy JSON Agreement Format

To maintain backward compatibility with already existing agreements, a **strategy pattern** is used to handle different agreement formats. The active format (JSON or eFLINT) is determined by the provider's validation config stored in etcd (`/policyEnforcer/configs/{provider}`).

| Agreement Format | Path to Layer 2 |
|-----------------|----------------|
| **eFLINT** | Used directly to construct the layered agreement rules |
| **Legacy JSON** | First translated to eFLINT phrases, then used to construct the layered agreement rules |

This means the three-layer architecture is always the canonical execution path — the strategy only differs in *how* Layer 2 is populated:

```
                     ┌─────────────────────────────┐
                     │       Validation Request      │
                     └──────────────┬──────────────┘
                                    │
                    ┌───────────────▼───────────────┐
                    │   Read provider config (etcd)  │
                    └───────────────┬───────────────┘
                                    │
               ┌────────────────────┴────────────────────┐
               │                                          │
    ┌──────────▼──────────┐                  ┌───────────▼──────────┐
    │  format = "eflint"  │                  │  format = "legacy"   │
    │                     │                  │                      │
    │  Use eFLINT phrases │                  │  Translate JSON →    │
    │  directly as        │                  │  eFLINT phrases      │
    │  Layer 2            │                  │  then use as Layer 2 │
    └──────────┬──────────┘                  └───────────┬──────────┘
               │                                          │
               └─────────────────┬────────────────────────┘
                                  │
                    ┌─────────────▼─────────────┐
                    │  Three-layer eFLINT eval   │
                    │  (Layer 1 + 2 + 3)         │
                    └───────────────────────────┘
```

The JSON-to-eFLINT translation maps the existing agreement structure (see `LegacyValidationStrategy`) onto the fact declarations described in [Layer 2: Agreement Rules](#layer-2-agreement-rules-02_agreement_ruleseflint) above — for example, the `relations` map becomes `has-relation` facts, `allowedArchetypes` becomes `relation-allows-archetype` facts, and so on.

### Flow-to-Normative Derivation Mapping

| Flow step | eFLINT construct | 
|-----------|-----------------|
| Agreement exists? | `agreement(data-steward)` |
| Steward archetypes / providers | `steward-supports-*` |
| User in relations? | `has-relation(req, steward)` |
| Relation allowances | `relation-allows-*` |
| Match archetypes | `valid-archetype` (conjunction) |
| Store compute providers | `valid-compute-provider` |
| Add to Valid/InvalidDataproviders | `permitted-at-steward` |
| ValidDataproviders > 0? | `permitted-request` |

### Scenario Walkthrough: Request Approval

A full end-to-end example. Requester **Niels** requests data from two stewards, **VU** and **UVA**.

**Request message:**
```protobuf
RequestApproval {
  type: "requestApproval"
  user: { id: "GUID", user_name: "Niels" }
  data_providers: ["VU", "UVA"]
  destination_queue: "policyEnforcer-in"
}
```

**Step 1 — Populate Request Facts (Layer 3):**
```eflint
+requester(Niels).
+requested-steward(VU).
+requested-steward(UVA).
```

**Step 2 — Overall Admissibility Query:**
```eflint
?Holds(permitted-request(Niels)).
// Response: query successful
```

**Step 3 — Per-Steward Admissibility:**
```eflint
?Holds(permitted-at-steward(Niels, VU)).
// Response: query successful

?Holds(permitted-at-steward(Niels, UVA)).
// Response: query successful
```

**Step 4 — Archetype Discovery:**
```eflint
?-archetype When valid-archetype(Niels, VU, archetype).
// Response: archetype("ComputeToData")

?-archetype When valid-archetype(Niels, UVA, archetype).
// Response: archetype("ComputeToData")
//           archetype("DataThroughTtp")
```

VU yields one archetype (Niels's relation only allows `ComputeToData`); UVA yields two because the intersection of Niels's allowances with UVA's supported archetypes is larger.

**Step 5 — Compute-Provider Discovery:**
```eflint
?-compute-provider When valid-compute-provider(Niels, VU, compute-provider).
// Response: compute-provider("SURF")

?-compute-provider When valid-compute-provider(Niels, UVA, compute-provider).
// Response: compute-provider("SURF")
```

**Step 6 — Action Trigger and Duty Creation:**

After a positive decision, the eFLINT Connector fires the `submit-data-request` act for each valid steward. This creates the `data-request` fact and the `obligated-log` duty:

```eflint
submit-data-request(Niels, VU). 
// Response: +data-request(requester("Niels"),data-steward("VU")) 
// +obligated-log(data-steward("VU"),requester("Niels"))

submit-data-request(Niels, UVA).
// Response: +data-request(requester("Niels"),data-steward("UVA")) 
// +obligated-log(data-steward("UVA"),requester("Niels"))
```

The `obligated-log` duty embodies the Hohfeldian power-duty relation: by exercising the power to submit a data request, the requester places the data steward under an obligation to log the processing of their data.

**Step 7 — Continuation Verification:**
```eflint
?Holds(permitted-continuation(Niels, VU, ComputeToData)).
// Response: query successful

?Holds(permitted-continuation(Niels, UVA, ComputeToData)).
// Response: query successful
```

**Resulting `ValidationResponse`:**
```protobuf
ValidationResponse {
  type: "validationResponse"
  request_type: "sqlDataRequest"
  valid_dataproviders: {
    "VU": {
      archetypes: ["computeToData"]
      compute_providers: ["SURF"]
    }
    "UVA": {
      archetypes: ["computeToData", "dataThroughTtp"]
      compute_providers: ["SURF"]
    }
  }
  invalid_dataproviders: []
  request_approved: true
  valid_archetypes: {
    user_name: "Niels"
    archetypes_per_steward: {
      "VU": ["computeToData"]
      "UVA": ["computeToData", "dataThroughTtp"]
    }
  }
}
```

The `ValidationResponse` is structurally identical to what the mock enforcer would produce for the same input (N2). The difference is that the decision is now backed by a formal normative derivation, and the Audit Logger can persist the derivation tree alongside the decision (F7).
